# Skill: code-review

Ревью кода перед коммитом. Вызывай этот скилл командой `/code-review` перед каждым коммитом.

## Что проверять

### 1. Git diff
Запусти `git diff` и `git diff --cached`, чтобы увидеть все изменения.

### 2. Корректность
- Нет ли опечаток в словарных данных (словенские слова, переводы, транскрипции)
- Логика работает как ожидается
- Нет сломанных ссылок на функции, переменные, CSS-классы

### 3. Качество кода
- Нет дублирования — не добавлено то, что уже есть
- Нет мёртвого кода (закомментированный код, неиспользуемые переменные)
- Нет console.log/debug-вывода, который не нужен в продакшне
- Изменения минимальны — не затрагивают то, чего не требовалось менять

### 4. PWA-специфика (этот проект)
- `sw.js`: если добавлены новые файлы — они добавлены в список кэша? Если нет — это нормально (API не кэшируется намеренно)
- `manifest.json`: иконки и метаданные не сломаны
- Нет хардкода путей, которые сломают работу оффлайн

### 5. UX
- Новые элементы интерфейса адаптивны (мобиль / десктоп)
- Специальные символы (š, č, ž, ...) отображаются корректно
- Панель символов и поле ввода не перекрывают друг друга

### 6. Данные уроков
- Структура объекта урока соответствует схеме остальных уроков
- Уровень (A1/A2/...) указан корректно
- Нет пустых полей там, где они обязательны

## Формат вывода

Выведи краткий отчёт:

```
РЕВЬЮ ИЗМЕНЕНИЙ
===============
Файлы: <список изменённых файлов>

OK:
- <что в порядке>

ПРОБЛЕМЫ:
- <проблема> → <как исправить>

ИТОГ: Готов к коммиту / Нужны правки
```

Если проблем нет — скажи "Готов к коммиту" и предложи сообщение коммита в стиле `fix:` / `feat:` / `refactor:`.
